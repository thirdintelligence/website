How It Works
Karpathy — Agentic Engineering (Sequoia AI Ascent 2026)
"Agentic engineering is about preserving the quality bar of professional software. You're still responsible for your software just as before, but can you go faster?"
Every agent in this system follows a structured 8-phase loop — from loading context and routing the task, through skill execution and quality checks, to logging what it learned and proposing improvements. The result: every session makes the next one better.
What makes it agentic: Task routing classifies every request by domain, complexity, and client. The right superskills activate automatically. Quality is checked against business rules. If something fails, the system self-heals or escalates. Every session's learnings are saved to memory and reviewed for patterns. When patterns repeat 3+ times, the system proposes its own workflow upgrades — meaning the architecture improves itself.
The goal: Build a system where every project compounds intelligence. A film made for Shaw teaches the system techniques that improve the next film for bkWatch. A website built for one client accelerates the next. Strategy insights from one engagement sharpen positioning across all clients. Nothing is wasted. Everything learns.
Tools: os.html is the live operational cockpit (you're looking at it) — the interface that displays state and routes work. Google Sheets holds canonical financial data. Gmail and Calendar provide communication context. The Obsidian vault visualizes the full memory graph. The OS is separate from the workflow itself — the workflow (memory, skills, learning, self-improvement) is the engine that runs behind this interface.

Memory System
Karpathy — LLM Wiki Pattern (Apr 2026)
"Synthesis beats retrieval for AI memory systems. The knowledge gets compiled once. Then it stays compiled." — Instead of retrieving from raw documents at query time, the agent incrementally builds and maintains a persistent wiki of structured, interlinked markdown files.
Third i uses 4 types of memory, modeled after how human cognition works. Each type serves a different purpose, and together they give every agent persistent intelligence across sessions — the agent never starts from scratch.
📘
Semantic
Facts, identity, brand DNA, client profiles. What things are. Rarely changes.
📋
Procedural
How to do things. Processes, workflows, rules, best practices. Evolves with upgrades.
📓
Episodic
Session logs, event history. What happened and what was learned. Patterns over time.
📌
Working
Active state right now. Current plan, open threads, priorities and blockers.
How it loads: Circle 1 memory (OS + semantic + procedural + working + persona) loads at every session start — the agent always knows who it is and what it's doing. Circle 2 memory (departments, episodic, techniques, client-specific) loads on-demand when the task needs it. Circle 3 memory is privacy-siloed — only one client's data loads at a time, never mixed.
How it mirrors the architecture: Every workflow and project in the infographic above has a matching memory folder. EXEC has department memory and R&D research. COMMS has email/meeting archives. Each WEB and FILM project has its own memory subfolder. When a new client is added, the memory structure is automatically cloned to match — that's why the memory cards sit behind the project cards in the diagram.

"Your prompts are now programs that program the LLM. Remarkably, these prompts are written in English." — Context is everything. The richer the context window, the smarter the agent's decisions.
Every session begins with a context check (Phase 0) that loads memory, verifies freshness, and connects to live data sources. This gives the agent a complete picture before it does any work.

Planning — The 8-Phase Execution Loop
Karpathy — Goal-Driven Execution
"Things get better in plan mode." — For non-trivial tasks, the agent should read, plan, and wait for approval before making changes. Goal-driven execution means defining success criteria and looping until verified.
Every task — from a strategy question to a film production to a financial audit — flows through the same structured pipeline. This consistency is what makes the system reliable and self-improving.
0 · Context → 1 · Route → 2 · Execute → 3 · QC → 3.5 · Heal → 4 · HITL → 5 · Learn → 6 · Reflect → 7 · Deliver

Superskills
Karpathy-style superskill contract
A superskill is a small, routed instruction module with a narrow task lane, explicit context inputs, hard boundaries, verification steps, and a learning hook. It makes the agent think first, stay simple, change only what is needed, and loop until the result is verified.
1. FramingDefines the task lane, assumptions, success criteria, and when the skill should activate.
2. ContextNames the memory files, project files, live connectors, or templates the skill must read before acting.
3. BoundariesStates what the skill may do, what requires HITL, and what belongs to another agent or skill.
4. VerificationRequires checks, readbacks, rendered review, or source-of-truth reconciliation before completion.
Color-coded by route: DEFAULT skills are used by all agents, EXEC handles operations/strategy, COMMS handles email/calendar communication, WEB is project-specific web guidance, and FILM is project-specific film workflow guidance.
DEFAULT Skills Used By All Agents

Framing: verifies the agent is in the right repo, rules, and workflow before work begins. Context: AGENTS.md, workflow files, memory freshness, tool availability. Boundary: does not perform project work; it prepares the run. Verification: reports missing rules, stale memory, or blocked tools before execution.
task_router Phase 1route selection
Framing: classifies domain, client, risk, and complexity. Context: user request, project state, live OS data. Boundary: routes rather than solves. Verification: selected agent/skills match client silo and task type.

Framing: checks the result against rules, source-of-truth data, and user intent. Context: changed files, Sheet readbacks, OS state, project rules. Boundary: flags or fixes only scoped issues. Verification: requires syntax checks, readbacks, and business-rule consistency.
self_healer Phase 3.5low-risk repair
Framing: diagnoses failed checks and chooses a small repair. Context: error output, expected behavior, affected source. Boundary: only auto-fixes low-risk issues; high-risk changes escalate. Verification: reruns the failing check before returning.

Framing: decides what should become project-specific, company-general, or technique memory. Context: session outcome and changed source of truth. Boundary: does not rewrite rules without approval. Verification: memory updates match completed work and avoid client mixing.

Framing: looks for repeated patterns and workflow upgrades. Context: recent episodic entries, current blockers, ROI. Boundary: proposes improvements; does not silently change major rules. Verification: recommendations include expected value and approval path.

Learning — Continuous Self-Improvement
Karpathy — The Karpathy Loop (AutoResearch, Mar 2026)
"To get the most out of the tools available now, you have to remove yourself as the bottleneck. You need to take yourself outside the loop. You have to arrange things such that they're completely autonomous." — "Evals are one of the highest-leverage things any AI developer can build."
Learning isn't optional — it's built into every single run. After every task completes (Phases 5–6), the system automatically logs what happened, classifies the knowledge, and checks for patterns. This is what makes the system genuinely self-improving.
The learning loop:
  • 1. Log — Every session's results are logged to episodic memory (what happened, what was learned)
  • 2. Classify — Knowledge is tagged: project-specific, company-general, or technique (cross-project)
  • 3. Detect — The reflector reviews the last 5 episodic entries for recurring themes or patterns
  • 4. Propose — When a pattern repeats 3+ times, the system proposes a workflow or rule upgrade
  • 5. Promote — Approved upgrades are promoted to TECHNIQUES.md or workflow/rules files
  • 6. Compound — All future sessions automatically benefit from the upgrade
What improves: Not just business knowledge (client preferences, project techniques) — but also the workflow and agentic architecture itself. If the system detects that a certain prompt pattern works better, or that a particular QC check catches more errors, it proposes upgrading its own instructions. The agent literally rewrites its own playbook — with human approval at the gate.
Self-QC Loop (Every Output)
  • 1. Generate output
  • 2. Run internal QC check against business rules + mantras
  • 3. If FAIL: self-diagnose, classify risk, auto-fix (low) or escalate (high)
  • 4. Log self-correction to episodic memory
